1. 单次重复 single-repeat

. . 重复上次更改，[count] 给出时覆盖原始计数。 'cpoptions' 包括 'y' 标志位时 ( cpo-y )，抽出 (yank) 操作也可重复。不重复命令行命令。 可用 . 重复简单的更改操作。[count] 省略时，沿用上次更改使用的计数。否则，覆 盖上次的计数，同时更新设置 v:count 和 v:count1 。 上次更改操作指定编号寄存器时，重复时寄存器编号自动递增。例见 redo-register 。 注意 重复选区操作时，选区范围 大小 与上次一致。参见 visual-repeat 。 @: @: 重复上一条命令行命令 [count] 次。 {仅当编译时加入 +cmdline_hist 特性才可用}

2. 批量重复 multi-repeat

:g :global E148 :[range]g[lobal]/{pattern}/[cmd] 在 [range] 范围内匹配模式 {pattern} 的各行执行 Ex 命令 [cmd] (缺省命令是 :p )。 :[range]g[lobal]!/{pattern}/[cmd] 在 [range] 范围内 不 匹配模式 {pattern} 的各行执行 Ex 命令 [cmd] (缺省命令是 :p )。 :v :vglobal :[range]v[global]/{pattern}/[cmd] 同 :g! 。 示例: :g/^Obsolete/d _ :d 之后使用下划线寄存器 ( quote_ ) 避免覆盖寄存器或剪贴板。速度也更快。 可用 '/' 之外的其他单字节字符作为 {pattern} 的定界符，但不可用字母、'\'、'"'、 '|' 或 '!'。这对包含 '/' 的搜索模式或替代字符串很有用。 模式语法见 pattern 。 注意 [cmd] 可自带范围；示例见 collapse 和 edit-paragraph-join 。 全局命令 (以上三条命令) 先扫描 [range] 范围内的行，标记匹配行 (多行匹配项仅标 记起始行)。第二次扫描时，定位光标到每个标记行，然后执行 [cmd] 命令。 :v 和 :g! 命令则对未标记行执行。一行被删除时，其标记也会自动消失。 [range] 缺省为整个缓冲区 (1,$)。按 CTRL-C 中断全局命令执行。单行执行过程中出 错仅中断该行，仍会继续执行下一标记行或未标记行。 E147 全局命令被递归调用时，仅作用于单行。此时不能给出范围。可用于找到所有匹配某一模 式但不匹配另一模式的行: :g/found/v/notfound/{cmd} 会先找到所有包含 "found" 的行，但仅当不匹配 "notfound" 时才执行 {cmd}。 可使用任意 Ex 命令，见 ex-cmd-index 。执行普通模式命令可用 :normal 命令: :g/pat/normal {commands} 必须确保 {commands} 以完整按键序列结束，否则 Vim 会逐行等待输入剩余按键。此时 屏幕不会刷新，因此不会有任何反馈。参见 :normal 。 撤销/重做命令会一次性处理整个全局命令。 前次上下文标记仅记录一次 ( '' 会回到全局命令开始前的光标位置)。 全局命令自动更新前次搜索模式和前次替代模式 (vi 兼容行为)，以便全局替换字符串: :g/pat/s//PAT/g 会将所有 "pat" 替换为 "PAT"。等价写法是: :%s/pat/PAT/g 可省下两个字符！ Ex 模式下执行全局命令时，可执行 :visual 命令，提供一种特殊用法。Vim 会依次 跳转到每个匹配行，进入普通模式，供用户执行普通模式命令，按 Q 返回 Ex 模式 后，再继续处理下一匹配行。此过程中不能递归调用全局命令。可按 CTRL-C 两次中断全 局命令执行。

3. 复杂重复 complex-repeat

q recording q{0-9a-zA-Z"} 在寄存器 {0-9a-zA-Z"} 里录制按键序列 (大写字母寄存器表 示追加键入内容到寄存器而非覆盖)。执行寄存器内容 (宏回 放)、映射和 :normal 命令时禁用本命令。 注意: 录制过程中，录制所用寄存器如果同时用于 y 和 p 命令，结果多半不符用户预期，因为 p 会插入寄存器 中录制的内容，而 y 会覆盖录制内容。 注意: 宏录制记录用户按键，而回放时这些按键被视同来自映 射。这一区别会影响某些行为，如撤销操作，仅手动输入命令 才会触发撤销同步。 q 停止宏录制 (实现细节备注: 停止录制 的 'q' 键不会存入寄 存器，除非该键来自映射)。 @ @{0-9a-z".=*+} 执行寄存器 {0-9a-z".=*+} 内的按键序列 [count] 次。注意 不能用寄存器 '%' (当前文件名) 和 '#' (轮换文件名)。 寄存器回放方式视同来自映射，也就是说，'wildchar' 和 'wildcharm' 的差异此处适用，撤销同步规则也与手动输入命 令不同。 "@=" 会提示输入表达式，其返回值被当作按键序列执行。 另见 @: 。 @@ E748 @@ 重复上次执行的 @{0-9a-z":*} 命令 [count] 次。 :@ :[addr]@{0-9a-z".=*+} 光标定位 [addr] 指定行 (缺省当前行) 后，将寄存器 {0-9a-z".=*+} 内容当作 Ex 命令执行。 'cpoptions' 里包含 'e' 标志位 ( cpo-e ) 时，寄存器内容 末行如果缺失 <CR> 会自动补上。 ":@=" 复用上次输入的表达式，其返回值被当作 Ex 命令执 行。 命令执行过程中，不触发映射。 行级寄存器中，行首带续行符 line-continuation (\) 的 行会与上一行合并。便于复制执行 Vim 脚本片段。 后续计划: 对地址范围内每一行分别执行寄存器内容。 :[addr]*{0-9a-z".=+} :star-compatible 'cpoptions' 里包含 '*' 标志位 cpo-star 时， :* 用法 等同 :@ 。非兼容模式 ('nocompatible') 缺省 不 开启。 'cpoptions' 里不含 '*" 标志位时， :* 为 :'<,'> 的别 名，选取前次可视选区作为范围 :star 。 :@: :[addr]@: 光标定位 [addr] 指定行 (缺省当前行) 后重复上次命令行。 :[addr]@ :@@ :[addr]@@ 光标定位 [addr] 指定行 (缺省当前行) 后重复上次执行 的 :@{register}。

其前导空白会被忽略。 示例: 多行写法 :set comments=sr:/*,mb:*,el:*/, \://, \b:#, \:%, \n:>, \fb:- 和以下单行写法等价: :set comments=sr:/*,mb:*,el:*/,://,b:#,:%,n:>,fb:- 反斜杠的前导空白字符会被忽略。注意: 被拆分行 (前一行) 行尾的空格不可随意添加； 能否保留空格由命令拆分位置决定。 需要空格时建议写在反斜杠之后。行尾空格不易察觉，易被误删。 :syn match Comment \ "very long regexp" \ keepend Vim9 脚本多数场景可省略续行反斜杠，但也有例外。见 vim9-line-continuation 。 使用 :append 和 :insert 命令时，有个问题: :1append \asdf . 此处反斜杠被看作续行符，因此等价于以下命令: :1appendasdf . 为解决此问题，可在 'cpoptions' 选项里临时包含 'C' 标志位 ( cpo-C ): :set cpo+=C :1append \asdf . :set cpo-=C 要 注意 函数内使用这两个命令时，需要在定义函数时提前加上 'C' 标志位，而不是在 函数执行时。以下是正确写法: :set cpo+=C :function Foo() :1append \asdf . :endfunction :set cpo-=C line-continuation-comment 要在续行与续行之间加上注释，行首要写 `"\ `。注意反斜杠后必须带空格。例如: let array = [ "\ 第一项注释 \ '第一项内容', "\ 第二项注释 \ '第二项内容', \ ] 原理: 多数程序用行尾反斜杠指示下行会继续。但 Vim 会因此和 Vi 不兼容。例如以 下 Vi 映射: :map xx asdf\ 因此，Vim 采用了这种特别的行首反斜杠续行方式。 续行一旦开始注释，则其后所有续行都会视为该注释的一部分。由于这一行为长 期如此，后来为支持在续行序列中途插入注释时，不能使用 \" ，因为它本身 已是合法的续行。最接近的替代写法是 `"\ `，虽然看起来仍有些古怪。要求在 反斜杠后加上空格，是为了尽量避免与普通注释行混淆。

5. Vim 插件包管理 packages

Vim 插件包是存放一个或多个插件的目录。和普通插件相比，有以下优点: - 插件包可作为归档下载并解压到独立目录。文件不与其他插件混杂。方便更新和卸载。 - 插件包可使用 git、mercurial 等版本仓库管理。更新便捷。 - 插件包可包含相互依赖的多个插件。 - 插件包可包含启动时自动加载的插件、以及按需通过 :packadd 手动加载的插件。 使用自动加载插件包 假定 Vim 文件位于 "~/.vim" 目录，要从 "/tmp/foopack.zip" 压缩包安装插件包: % mkdir -p ~/.vim/pack/foo % cd ~/.vim/pack/foo % unzip /tmp/foopack.zip 目录名 "foo" 为自定义插件名，可任意自选。 解压后 ~/.vim 下目录结构如下: pack/foo/README.txt pack/foo/start/foobar/plugin/foo.vim pack/foo/start/foobar/syntax/some.vim pack/foo/opt/foodebug/plugin/debugger.vim Vim 启动时，处理完 .vimrc 后，会扫描 'packpath' 下所有 "pack/*/start" 目录下的 插件。先将这些插件目录加入 'runtimepath'，然后加载其中全部脚本。两步处理的用途 详见 packload-two-steps 。 为了在 .vimrc 解析阶段就能调用插件包内功能， :colorscheme 和 autoload 都会 同时在 'runtimepath' 与 'packpath' 下检索。详见每项的对应文档。 此例中，Vim 会找到 "pack/foo/start/foobar/plugin/foo.vim" 并将 "~/.vim/pack/foo/start/foobar" 插件目录加入 'runtimepath'。 加载 "foobar" 插件后，假定它会将 'filetype' 设为 "some"，Vim 会找到 syntax/some.vim 文件，因为 'runtimepath' 里已包含相应目录。 同时加载 ftdetect 类型脚本 (如有)。 注意 "pack/foo/opt" 下的插件不会自动加载，启动时只自动加载 "pack/foo/start" 下 的。下文 pack-add 说明 "opt" 目录的用法。 关闭插件加载时，也会跳过插件包自动加载功能，见 load-plugins 。 要提前加载软件包以更新 'runtimepath': :packloadall 该命令不受插件加载开关限制。自动加载流程则只会执行一次。 插件内有 "after" 目录时，该后置目录会自动追加到 'runtimepath' 的末尾，其中脚本 晚于其他脚本加载。 使用自动加载的单个插件 如果只有单个插件，而没有完整插件包目录结构，需自行补全目录结构: % mkdir -p ~/.vim/pack/foo/start/foobar % cd ~/.vim/pack/foo/start/foobar % unzip /tmp/someplugin.zip 现在目录结构如下: pack/foo/start/foobar/plugin/foo.vim pack/foo/start/foobar/syntax/some.vim 此后，就可以正常使用上述的自动加载机制。 可选插件 pack-add 要从软件包里手动加载可选插件，可用 :packadd 命令: :packadd foodebug 该命令会扫描 'packpath' 下所有 "pack/*/opt/foodebug" 插件，并加载其中找到的 ~/.vim/pack/foo/opt/foodebug/plugin/debugger.vim 脚本。 适用于按条件加载插件。如取决于 Vim 是否支持某特性或是否缺少某依赖。 启动时可在 .vimrc 里加载可选插件: :packadd! foodebug 加 "!" 时只注册插件目录，确保 Vim 以 --noplugin 启动时不加载任何插件。 插件包仅包含 "opt" 目录完全正常。此时仅在需要特定插件时才会加载。 文件存放规范 由于 :colorscheme 会同时在 "pack/*/start" 和 "pack/*/opt" 下搜索色彩方案，因 此可放在任一位置。推荐放在 "pack/*/opt" 下，例如 ".vim/pack/mycolors/opt/dark/colors/very_dark.vim"。 文件类型插件则应在 "pack/*/start" 下，以确保始终能找到。只有同一文件类型有多个 插件，并希望按条件加载 (通过 :packadd ) 时，才适合放在 "pack/*/opt" 下。例如 取决于编译器版本: if foo_compiler_version > 34 packadd foo_new else packadd foo_old endif 插件包内不推荐使用 "after" 目录。但不禁止。

6. Vim 插件包制作 package-create

本节适用于开发一到多个插件并打包分发的插件开发者。 对于两个无关的插件，建议拆分成两个插件包，以便 Vim 用户按需选用。但也可整合为 包含多个可选插件的单个插件包，告知用户用 :packadd 手动选用。 插件包分发有多种方案。可通过归档压缩包，也可通过版本仓库。归档适合更多用户，但 版本更新繁琐。而版本仓库更新便捷，但用户需要安装 "git" 之类的工具。也可同时采 用，github 发布新版本时可自动生成归档压缩包。 目录布局应为: start/foobar/plugin/foo.vim " 自动加载，定义命令 start/foobar/plugin/bar.vim " 自动加载，定义命令 start/foobar/autoload/foo.vim " 延迟加载，调用 foo 命令时载入 start/foobar/doc/foo.txt " foo.vim 帮助文档 start/foobar/doc/tags " 帮助标签文件 start/foobar/lang/<lang_id>/LC_MESSAGES/foobar.mo " 插件消息的 <lang_id> 语言版本。可选 opt/fooextra/plugin/extra.vim " 可选插件，定义命令 opt/fooextra/autoload/extra.vim " 延迟加载，调用 extra 命令时载入 opt/fooextra/doc/extra.txt " extra.vim 帮助文档 opt/fooextra/doc/tags " 帮助标签文件 此时用户可进行以下操作: mkdir ~/.vim/pack cd ~/.vim/pack git clone https://github.com/you/foobar.git myfoobar "myfoobar" 是用户自选插件包名，只需不和其他包重名即可。 帮助文档需解释插件用法并告知用户如何手动加载可选插件: :packadd! fooextra 也可将该命令加入主插件脚本，实现按需自动启用可选插件。 package-doc package-documentation 可用 :helptags 命令自动生成 doc/tags 文件。打包时一并放入，用户在 pack 目录 下安装插件包后可直接查看帮助。修改插件帮助后必须重新生成标签文件: :helptags path/start/foobar/doc :helptags path/opt/fooextra/doc package-translation 插件要能显示多语言翻译消息，需完成以下几个步骤。 首先，通过 bindtextdomain() 函数定义包名和翻译所在根目录: :call bindtextdomain("foobar", \ fnamemodify(expand("<script>"), ':p:h') .. 7. 脚本调试 debug-scripts
